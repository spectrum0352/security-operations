## Integrations with 3<sup>rd</sup> party

### Integration with AWS

This document outlines the process of integrating Microsoft Defender for
Cloud (MDC) with Amazon Web Services (AWS), providing a comprehensive
guide to setup, configuration, and benefits.

I. Authentication and Authorization:

MDC requires access to your AWS environment to assess security posture
and provide recommendations. This is achieved through secure
authentication and authorization using AWS Identity and Access
Management (IAM):

- **IAM Role:** The recommended approach. Create an IAM role in your AWS
  account with permissions specifically tailored for MDC. This role
  grants MDC read-only access to the necessary AWS resources, adhering
  to the principle of least privilege. This is the preferred method as
  it avoids managing long-lived credentials.

- **External ID:** When creating the IAM Role, you will need to provide
  an External ID. This External ID is provided by Microsoft Defender for
  Cloud and acts as a security measure to prevent confused deputy
  attacks. It ensures that only MDC can assume this role.

II\. AWS Configuration:

Before connecting MDC, ensure the following AWS services are enabled:

- **AWS Config:** Enable AWS Config to record configuration changes for
  your AWS resources. This provides MDC with a history of changes,
  crucial for identifying misconfigurations and compliance deviations.

- **AWS Security Hub:** Enable AWS Security Hub to aggregate, organize,
  and prioritize security alerts from various AWS services. MDC
  integrates with Security Hub to gain a consolidated view of your AWS
  security posture.

III\. MDC Configuration:

1.  **Add AWS Environment:** In the MDC portal, add your AWS account as
    a new environment. This involves providing the AWS account ID and
    configuring the connection using the IAM role you created. You will
    be prompted to enter the External ID provided by MDC when
    configuring the trust relationship with the IAM Role.

2.  **Select Defender Plans:** Choose the appropriate Defender plans
    based on your security requirements. These plans offer different
    levels of protection and include:

    - **Foundational CSPM (Cloud Security Posture Management):**
      Provides basic security posture assessments and recommendations.

    - **Defender CSPM:** Offers advanced CSPM capabilities, including
      threat detection, vulnerability assessment, and compliance
      monitoring.

    - **Cloud Workload Protection (CWP):** Protects your EC2 instances
      and other compute resources with features like vulnerability
      assessment, file integrity monitoring, and adaptive threat
      protection.

IV\. Data Flow and Monitoring:

Once the integration is configured, the following data flow occurs:

1.  **API Queries:** MDC uses the assumed IAM role to make read-only API
    calls to AWS services. These calls retrieve configuration data and
    security events for analysis.

2.  **CloudTrail (Optional):** If enabled in your AWS account,
    CloudTrail logs these API calls. This provides an audit trail of
    MDC's actions within your AWS environment.

3.  **Security Hub Integration:** MDC integrates with AWS Security Hub
    to receive security findings and alerts generated by various AWS
    services.

4.  **Optional Agents for Deeper Visibility:** For enhanced security
    monitoring and remediation capabilities on EC2 instances, consider
    deploying the following agents:

    - **Azure Arc Agent:** Connects your EC2 instances to Azure,
      enabling centralized management and monitoring. This is required
      for many of the advanced features in MDC.

    - **Log Analytics Agent (MMA/AMA):** Collects security logs and
      events from your EC2 instances and sends them to MDC for analysis.
      The Azure Monitoring Agent (AMA) is the recommended agent.

    - **(Consider) Telegraf Agent:** A flexible agent for collecting
      metrics and logs from various sources, which can be integrated
      with Azure Monitor.

V. Benefits of Integration:

- **Improved Security Posture:** Identify and mitigate security
  vulnerabilities and misconfigurations in your AWS environment.

- **Centralized Security Management:** Manage security across your AWS
  environment and other cloud platforms from a single pane of glass in
  MDC.

- **Proactive Threat Detection:** Detect and respond to threats in
  real-time using MDC's advanced threat intelligence.

- **Compliance Monitoring:** Ensure your AWS environment complies with
  industry regulations and best practices.

VI\. Step-by-Step Integration Summary:

1.  **AWS Console:** Create an IAM role with the appropriate permissions
    for MDC and note the External ID provided by MDC.

2.  **AWS Console:** Enable AWS Config and AWS Security Hub.

3.  **MDC Portal:** Add your AWS account as an environment, providing
    the account ID and using the IAM role you created. Enter the
    External ID.

4.  **MDC Portal:** Select the desired Defender plans.

5.  **EC2 Instances (Optional):** Onboard EC2 instances to Azure Arc and
    install the Log Analytics agent for deeper visibility.

VII\. Key Considerations:

- **Least Privilege:** Grant MDC only the necessary permissions required
  for its functionality.

- **Regular Audits:** Regularly review the permissions granted to the
  MDC IAM role.

- **External ID:** Always use the External ID provided by MDC when
  configuring the IAM role trust relationship.

- **Agent Deployment:** Carefully plan the deployment of agents on EC2
  instances to minimize performance impact.

- **Cost Optimization:** Choose the Defender plans that align with your
  security needs and budget.

By following these steps, you can effectively integrate AWS with MDC,
enhancing your cloud security posture and gaining centralized visibility
into your AWS environment.

**Connecting AWS to Microsoft Defender for Cloud:**

1.  **AWS Console:** Create an IAM role for Security Center.

2.  **AWS Console:** Create an AWS user for Security Center.

3.  **Microsoft Defender for Cloud:** Connect AWS account using the
    created IAM role and user credentials.

4.  **Initial Steps:** Enable AWS Config, enable AWS Security Hub, and
    verify data flow to Security Hub.

**Connecting Cloud Environments to MDC:**

- **AWS:**

  1.  Create an IAM role for Security Center on the AWS console. This
      connects AWS to MDC.

  2.  Create an AWS user for Security Center to authenticate MDC to AWS.

  3.  Enable AWS Config and AWS Security Hub.

  4.  Verify data flow to Security Hub.

### Integration with GCP

This document outlines the process of integrating Google Cloud Platform
(GCP) with Microsoft Defender for Cloud (MDC) to enhance security
posture by centralizing security monitoring and threat detection across
multiple cloud environments.

The integration process involves the following key steps:

1.  **Configure GCP Security Command Center:**

    - Enable the Security Health Analytics and its API. This is crucial
      for MDC to receive security findings from GCP.

    - Ensure that the necessary security findings are enabled within
      Security Health Analytics. MDC relies on these findings to provide
      insights and recommendations. Common findings include
      vulnerabilities, misconfigurations, and malware detections.

2.  **Establish the Connection (Cloud Connectors):**

    - In the Microsoft Defender for Cloud portal, navigate to the
      "Environment settings" section.

    - Select "GCP" and follow the prompts to add a new connector. This
      establishes the link between MDC and your GCP project. You will
      need appropriate permissions in both environments.

3.  **Create Service Account and Key (GCP):**

    - In the GCP console, navigate to IAM & Admin -\> Service Accounts.

    - Create a dedicated service account specifically for the MDC
      integration. Give it a descriptive name (e.g., "MDC-Connector").

    - Grant the service account the necessary roles. The "Security
      Center Admin" role (roles/securitycenter.admin) is typically
      required to allow MDC to access Security Command Center data.
      Review Google's documentation for the most up-to-date role
      requirements.

    - Create a private key (JSON format) for the service account.
      **Store this key securely.** It will be used by the MDC connector
      to authenticate with GCP. Download the key file.

4.  **Configure the MDC Connector with the Service Account Key:**

    - During the connector setup in the MDC portal, you will be prompted
      to upload the JSON key file you downloaded in the previous step.
      This allows MDC to authenticate as the service account and access
      the necessary GCP data.

5.  **Enable the Security Command Center API (if not already enabled):**

    - In the GCP console, navigate to the API Library.

    - Search for "Security Command Center API" and enable it for the
      project you are connecting to MDC. This facilitates the data flow
      from GCP to MDC. While enabling Security Health Analytics usually
      enables this, it's good to double-check.

Data Transfer from GCP to MDC:

The Security Command Center API facilitates the transfer of security
findings and events from GCP to MDC. MDC then analyses this data to
provide security assessments, recommendations, and threat detection
capabilities.

**Categorization:**

- **GCP Integration:**

  - Configuring Security Command Center (including enabling required
    findings)

  - Creating and configuring a dedicated Service Account and Key

  - Adding Cloud Connectors in MDC (using the Service Account Key)

  - Enabling the Security Command Center API (verification)

- **Azure Arc and Defender (Separate Topic):**

  - Onboarding Servers to Azure Arc

  - Installing the Log Analytics Agent

**Additional Considerations:**

- **Data Privacy and Compliance:** Ensure the data transfer and storage
  comply with all relevant regulations (e.g., GDPR, HIPAA, CCPA).
  Understand what data is being shared and how it is being used.

- **Security Best Practices:** Protect the service account key. Use
  least privilege principles when assigning roles to the service
  account. Regularly review the integration configuration.

- **Monitoring and Alerting:** Configure monitoring and alerting in both
  GCP and MDC to detect any issues with the integration or security
  threats. MDC provides its own alerting capabilities, but you can also
  integrate with other systems.

- **Regular Review and Updates:** Keep the integration configuration
  up-to-date with the latest security recommendations from both
  Microsoft and Google. Regularly review the permissions and roles
  assigned to the service account.

- **Cost Optimization:** Be aware of the potential costs associated with
  using Security Command Center and data transfer. Optimize the
  configuration to ensure you are only collecting the necessary data.

- **Troubleshooting:** Refer to the official Microsoft and Google
  documentation for troubleshooting tips if you encounter any issues
  during the integration process. Check the logs in both platforms for
  error messages.

By following these detailed steps and considering the additional points,
organizations can effectively integrate GCP with MDC to enhance the
security of their multi-cloud environments.

**Connecting Cloud Environments to MDC:**

- **GCP:**

  1.  Configure GCP Security Command Center, enabling Security Health
      Analytics and its API.

  2.  Add cloud connectors from MDC.

  3.  Create dedicated service accounts and a private key on the GCP
      console.

  4.  Enable the Google Security Command Center API in the Google Cloud
      console API library for the relevant project.

**Connecting AWS to Defender for Cloud:**

1.  **AWS Side:** Create an IAM role for Security Center.

2.  **Azure Side:** Connect your AWS account to Defender for Cloud using
    the created IAM role's ARN. This allows Defender for Cloud to
    authenticate to AWS.

**Initial Stages of Connecting AWS to Defender for Cloud:**

- Enable AWS Config.

- Enable AWS Security Hub.

- Verify data flow to Security Hub.

**Connecting GCP to Defender for Cloud:**

1.  **GCP Side:** Configure GCP Security Command Center, enable Security
    Health Analytics and its API. Create a dedicated service account
    with a private key.

2.  **Azure Side:** Add a cloud connector in Defender for Cloud, using
    the GCP service account and private key for authentication.

**Facilitating Data Transfer from GCP to Defender for Cloud:**

1.  In the Google Cloud Console, go to the API Library.

2.  Select the project you want to connect to Defender for Cloud.

3.  Enable the "Security Command Center API."

**Protecting GCP and AWS VMs with Defender for Cloud:**

1.  **Onboarding:**

    - Use Azure Arc for Servers to connect the VMs to Azure.

    - Install the Azure Monitor Agent (formerly Log Analytics agent) on
      the servers. This makes them visible in Defender for Cloud.

**Connecting GCP to Microsoft Defender for Cloud:**

1.  **GCP Console:** Configure GCP Security Command Center and enable
    Security Health Analytics API.

2.  **Microsoft Defender for Cloud:** Add cloud connectors.

3.  **GCP Console:** Create a dedicated service account and private key.

**Facilitating Data Transfer from GCP to Microsoft Defender for Cloud:**

1.  **GCP Console:** Go to the API Library.

2.  Select the project you want to connect to Microsoft Defender for
    Cloud.

3.  Enable the "Security Command Center API."

**Protecting GCP and AWS VMs with Microsoft Defender for Cloud:**

1.  Enable Microsoft Defender for Cloud on your Azure subscription.

2.  **Permissions:** Owner on the GCP organization/project and
    Contributor on the AWS account.

3.  Install the Azure Arc agent on each GCP/AWS VM.

4.  Install the Log Analytics agent on each GCP/AWS VM. *Both* agents
    are required.

5.  Azure Arc agent establishes the connection.

6.  Log Analytics agent collects security data.

7.  Defender for Cloud analyzes the data.

### Integration with Sentinel

This document outlines the integration of Microsoft Defender for Cloud
(MDC) with Microsoft Sentinel for Security Information and Event
Management (SIEM). This integration allows you to leverage Sentinel's
advanced analytics and threat intelligence capabilities to analyze
security alerts generated by MDC.

The integration process involves configuring MDC to send its security
alerts to Sentinel. This can be broken down into the following steps:

**1. Enabling the Stream of Microsoft Defender for Cloud Alerts to
Sentinel:**

This is the core process and involves configuration on both the MDC and
Sentinel sides.

**2. Microsoft Defender for Cloud (MDC) Configuration:**

- Within the Azure portal, navigate to Microsoft Defender for Cloud.

- Locate the "Environment settings" or similar section related to data
  export/integration. (The exact location may vary slightly depending on
  the MDC interface version).

- Select the relevant subscription(s) for which you want to send alerts
  to Sentinel.

- Find the option to integrate with Log Analytics workspaces. Sentinel
  uses Log Analytics as its underlying data store.

- Choose the appropriate Log Analytics workspace where you want to send
  MDC alerts. This workspace should be the same one used by your
  Sentinel instance.

- Enable the streaming of security alerts to the selected Log Analytics
  workspace. You might have options to select specific alert severities
  or types.

**3. Microsoft Sentinel Configuration:**

- In the Azure portal, navigate to Microsoft Sentinel.

- Select the workspace that is associated with the Log Analytics
  workspace you configured in the MDC steps.

- Confirm that the connection between Sentinel and the Log Analytics
  workspace is active. MDC alerts should automatically start appearing
  in Sentinel after the MDC configuration is complete. You can verify
  this by checking the "Logs" section in Sentinel and querying the
  tables that store MDC alerts (e.g., SecurityAlert).

**4. PowerShell Automation (Optional but Recommended):**

While the above steps can be performed through the Azure portal, using
PowerShell can automate and streamline the process, especially for
managing multiple subscriptions. The provided link to the GitHub
repository (Microsoft-Defender-for-Cloud/Powershell scripts/3rd party
SIEM integration at main · Azure/Microsoft-Defender-for-Cloud · GitHub)
contains scripts that can assist with this. Look for scripts
specifically related to Sentinel integration within the "3rd party SIEM
integration" folder. These scripts may handle tasks like:

- Automating the connection between MDC and Log Analytics.

- Configuring alert streaming.

- Potentially setting up specific alert rules or playbooks within
  Sentinel.

**Important Considerations:**

- **Log Analytics Workspace:** Ensure that both MDC and Sentinel are
  connected to the *same* Log Analytics workspace. This is crucial for
  the integration to function correctly.

- **Permissions:** Appropriate Azure permissions are required to
  configure both MDC and Sentinel. You'll need permissions to modify MDC
  settings, link workspaces, and configure data connectors in Sentinel.

- **Alert Mapping:** Understand how MDC alerts are mapped to the Common
  Event Format (CEF) or other formats used by Sentinel. This will be
  important for writing queries and creating rules within Sentinel.

- **Cost:** Be aware of the potential costs associated with ingesting
  and storing logs in Log Analytics.

- **Troubleshooting:** If alerts are not appearing in Sentinel, check
  the connection status between MDC and the Log Analytics workspace.
  Also, review the logs in both MDC and Sentinel for any error messages.

By following these steps, you can successfully integrate Microsoft
Defender for Cloud with Microsoft Sentinel and enhance your security
monitoring capabilities.

### Integration with ServiceNow

Microsoft Defender for Cloud and ServiceNow Integration

- Allows for bi-directional data flow between the two platforms.

- Enables users to create and monitor tickets (incident, problem,
  change) in ServiceNow from Defender for Cloud.

- Requires CSPM enabled subscription and application registry setup in
  ServiceNow.

How to integrate MDC with ServiceNow?

1.  Create an application registry in ServiceNow.

2.  Navigate to MDC -\> Environment settings -\> ServiceNow Integration
    -\> Add integration.

3.  Provide details including name, scope, ServiceNow instance details
    (URL, username, password, client ID, client secret).

4.  Configure ticket data (incident, problem, change tables).

**Benefits of Integration**

- Improved efficiency in incident management.

- Streamlined communication between security and IT operations teams.

- Ability to prioritize remediation of security issues.

**Pricing**

- This is a premium feature requiring a CSPM paid plan.

### MDC REST API

**Microsoft Defender for Cloud REST APIs**

Microsoft Defender for Cloud offers a set of REST APIs that allow you to
programmatically interact with its security features and data. These
APIs provide a powerful way to automate security tasks, integrate with
other tools, and gain deeper insights into your security posture.

**Key Use Cases:**

1.  **Security Posture Assessment:**

    - Retrieve detailed information about security assessments,
      vulnerabilities, and threats.

    - Analyze security recommendations and generate custom reports.

    - Trigger security assessments on demand.

2.  **Incident Response and Threat Hunting:**

    - Query incident data, alerts, and security events.

    - Automate incident response workflows, such as creating tickets or
      notifying security teams.

    - Conduct advanced threat hunting and investigation using API-driven
      queries.

3.  **Security Policy Management:**

    - Create, modify, and delete security policies.

    - Enforce security policies across your Azure environment.

    - Monitor policy compliance and generate reports.

4.  **Integration with Other Tools:**

    - Integrate MDC with other security tools and platforms using APIs.

    - Build custom security dashboards and visualizations.

    - Automate security workflows and processes.

**Accessing the REST APIs:**

To access the MDC REST APIs, you will typically need:

- **Azure Subscription ID:** This identifies your Azure subscription.

- **Azure Resource Group:** This groups related Azure resources.

- **API Version:** This specifies the version of the API you want to
  use.

The base URL for API calls generally follows this format:

https://management.azure.com/subscriptions/{subscriptionId}/resourceGroups/{resourceGroupName}/providers/Microsoft.Security/{resourceName}?api-version={api-version}

**Important Considerations:**

- **Authentication and Authorization:** You'll need to authenticate your
  requests using Azure Active Directory (AAD) and authorize your
  application to access the necessary resources.

- **API Rate Limits:** Be aware of rate limits and potential throttling
  to avoid service disruptions.

- **API Versioning:** Use the latest stable API version to benefit from
  new features and improvements.

- **Error Handling:** Implement proper error handling mechanisms to
  gracefully handle API errors and exceptions.

**For more detailed information and specific API reference, please refer
to the official Microsoft documentation and GitHub repository:**

- **Microsoft Defender for Cloud REST API Documentation:**
  <https://learn.microsoft.com/en-us/rest/api/defenderforcloud/>

- **GitHub Repository:**
  <https://github.com/Azure/Microsoft-Defender-for-Cloud>

By leveraging the MDC REST APIs, you can enhance your security posture,
streamline operations, and gain valuable insights into your Azure
environment.
