Linux Log collection
By collecting, analyzing, and ingesting Linux logs into security tools like SIEM (Security Information and Event Management), XDR, or CSPM, you gain critical visibility into system activity. This allows you to identify anomalies, track user behavior, and respond to threats in real-time.

Core Logging Concepts To manage logs effectively, you must understand the infrastructure supporting them: • Log Levels (Severity): Linux uses standard levels to categorize importance: Debug, Info, Notice, Warning, Error, Critical, Alert, and Emergency. • Log Rotation: Managed by the logrotate utility. It prevents disk exhaustion by archiving old logs, compressing them, and deleting them after a set period. • Log Aggregation: The practice of sending logs from multiple endpoints to a central server (e.g., Elastic Stack, Splunk) for unified analysis. • Journald: Modern Linux distributions (using systemd) use journald to collect binary logs, which are viewed using the journalctl command.
Essential Linux Logs for Security A. Authentication and Authorization These are the highest-priority logs for detecting brute-force attacks and unauthorized access. • /var/log/auth.log (Debian/Ubuntu) or /var/log/secure (RHEL/CentOS): o Records all authentication attempts (SSH, local logins, sudo). o Security Value: Detects failed login spikes or unauthorized sudo attempts. • /var/log/lastlog: A binary file showing the last login time for every user. (View with lastlog command). • /var/run/utmp and /var/log/wtmp: o utmp shows users currently logged in. o wtmp maintains a history of all logins/logouts. (View with last command). • /var/log/btmp: Records all failed login attempts. (View with lastb command). B. System and Kernel Activity • /var/log/syslog (Debian) or /var/log/messages (RHEL): The "catch-all" log for general system messages and non-critical kernel info. • /var/log/kern.log: Specific to kernel events, hardware errors, and driver issues. • dmesg: The kernel ring buffer. It is vital for forensic analysis of hardware changes or kernel-level exploits immediately after boot. C. Security and Auditing • /var/log/audit/audit.log (auditd): o The most granular security log. It tracks system calls, file access (who touched which file), and network socket changes. o Security Value: Essential for compliance (PCI-DSS, SOC2) and deep forensics. • SELinux / AppArmor Logs: Usually found in audit.log or syslog. They record when a mandatory access control policy blocks a process. D. Network and Services • Firewall Logs: (e.g., /var/log/ufw.log or iptables output via syslog). Tracks dropped packets and connection attempts. • Web Server Logs: (/var/log/apache2/ or /var/log/nginx/). Tracks HTTP 404/500 errors and suspicious URL patterns (SQLi, XSS). • Cron Logs: (/var/log/cron or in syslog). Records scheduled tasks. Attackers often use Cron for persistence.
Log Collection & Management Methods
Method Description Logrotate Automates the rotation, compression, and removal of log files to save disk space. Rsyslog An enterprise-grade logging daemon that can filter logs and forward them over the network (UDP/TCP) to a SIEM. Fluentd / Logstash Data collectors used to "parse" logs into JSON before sending them to a centralized database. Journalctl The primary tool for querying systemd logs. Example: journalctl -u ssh to see only SSH service logs.

Best Practices for Security
Remote Logging: Always send logs to a remote server. If a hacker gains root access, the first thing they will do is delete the local logs to hide their tracks.
File Integrity Monitoring (FIM): Use tools like Wazuh or AIDE to ensure log files themselves haven't been tampered with.
Time Synchronization: Ensure all servers use NTP (Network Time Protocol). Accurate timestamps are mandatory for correlating events across different servers during an investigation.
Ingest into SIEM: Ensure you are alerting on specific patterns, such as multiple failed logins followed by a successful one (potential brute force).
