# Log Collection and SIEM Guide

## Objective

Log review is a critical part of security assurance. Relying on logs only after a major incident wastes the opportunity to identify threats and vulnerabilities early and to respond proactively. Because there are many log types and formats, organizations need systems that can aggregate and correlate data from different log sources to produce actionable intelligence.

---

## Log management

Log management is the process of collecting, storing, analyzing, and retaining log data from various sources within an IT infrastructure. It's a critical component of effective security and operations management.
Key Elements:
•	Log Sources: Logs can be generated from various sources, including network devices (routers, switches, firewalls), servers (Linux, Windows, Mac), cloud platforms (Azure, GCP, AWS), identity and access management (IAM) systems, and mobile devices.
•	Log Collection: Centralizing logs from different sources for analysis and correlation.
•	Log Storage: Storing logs securely, often with encryption for sensitive data.
•	Log Analysis: Analysing logs to identify trends, anomalies, security threats, and performance issues.
Best Practices:
•	Data Compatibility: Ensure that your log management solution can handle logs from various platforms.
•	Log Retention: Retain logs for a specified period based on compliance requirements and business needs.
•	Security: Protect sensitive data in stored logs and ensure secure log shipping.
•	Cloud Services: Configure diagnostic and threat detection logs for PaaS services like storage accounts and databases.
•	Log Shipping: Use tools like Rsyslog, Windows Event Log Forwarding, or third-party solutions to securely ship logs for analysis, storage, and archiving.
By implementing effective log management practices, organizations can gain valuable insights into their IT infrastructure, improve security, and optimize operations.


## 1. SIEM Deployment

Not all security incidents are revealed by a single event. Taken together, events that look valid and proper on their own can reveal a security problem.

**Example:** VPN logs show that User-1, a sales representative who regularly travels to Asia, has logged in from Beijing. Moments later, the RFID physical security system shows User-1 badging in at the corporate office in Downers Grove, IL. Neither event alone is anomalous, but correlated together they indicate a security problem.

A **security information and event management (SIEM)** solution provides real-time or near-real-time analysis of logs and alerts generated by network hardware and applications. SIEM expands insight into intrusion detection and prevention by aggregating and correlating security intelligence. SIEM can be deployed as software, a hardware appliance, or an outsourced managed service.

Effective SIEM deployment should include:

- Logging all relevant events without being cluttered by irrelevant data
- Clearly documenting the scope of monitored events
- Developing use cases that define exactly what does and does not count as a threat
- A response plan for when a threat alert fires
- A robust ticketing process to track flagged events
- Scheduled, regular threat hunting to catch anything alerts missed
- An evidence trail to support auditors and forensic analysts

## 2. Security Data Collection and Use Cases

Intelligence generally loses value over time, so data captured and analyzed in real time or near-real time is most valuable — sometimes valuable enough to limit or avoid attack damage entirely. Gathering and analyzing security intelligence involves several labor-intensive steps: identifying relevant data, collecting it, transforming it into a usable form, aggregating and correlating sources, analyzing the correlated data for significant patterns, and deciding what action to take.

SIEMs automate much of this cycle — especially collection and processing — producing actionable insight faster than manual or piecemeal log review. They can also automate parts of analysis, reporting, and dissemination.

Careful planning at the front end reduces later analysis work. Deciding what to collect, based on your security and compliance requirements, is itself a front-end analysis step that saves significant effort later. A SIEM *could* ingest every log across your systems, but this isn't advisable: too much data creates network overhead, obscures the events that matter, and adds unnecessary analysis work. Configure the SIEM to focus on the events tied to the security and compliance risks you've already identified through risk management.

Early SIEMs were difficult to configure, limited in capability, and required significant expertise to use well. Some organizations found they simply added noise — another alert source without added insight. All alerting systems face a tradeoff between **false negatives** (threats going unnoticed) and **false positives** (alert volume overwhelming analysts). To manage this tradeoff, a successful SIEM deployment depends on well-built **use cases**.

A use case is a specific, defined condition that should be reported — for example, a suspicious logon to a high-value asset by a privileged account, or a process executing from an administrative share. A use case template specifies the data sources containing relevant indicators, the query logic used to correlate them, and the actions a detected event should trigger. Use cases come out of threat modeling, and each one should capture:

- **When** the event started (and ended, if relevant)
- **Who** was involved
- **What** happened, in enough detail to distinguish it from other events
- **Where** it happened and where it originated — e.g., the host, file system, or network port involved, and whether the session came from an external IP over VPN

## 3. Security Data Normalization

Security data arrives from many different sources, and in raw form it isn't always usable. To surface patterns or anomalies, data often needs to be reformatted or restructured — whether it will be reviewed by people or software.

SIEMs typically collect data using one or more of these methods:

- **Agent-based** — An agent runs on each host, filtering, aggregating, and normalizing log data locally before sending it to the SIEM. Examples include Elastic's Beats agents (for events and application logs) and OSSEC (for intrusion detection data).
- **Listener/collector** — Hosts push updates to the SIEM server via a protocol such as syslog or SNMP, with a process on the management server parsing and normalizing each source.
- **Sensor** — In addition to logs, the SIEM can capture packet captures and traffic flow data from sniffers deployed at strategic network points, forwarding that data back to the central management instance.

In a typical deployment, the SIEM aggregates network traffic from SPAN (port-mirroring) and TAP sensors placed at key network points, endpoint data from IDS/EDR agents on client workstations, and application/access log data via agents or syslog.

## 4. Log Collection Ownership: Shared Responsibility

Log collection is typically a shared responsibility between IT/Cloud Operations and Security, split by stage of the logging lifecycle.

### IT / Cloud / Platform Operations
Responsible for generating and forwarding logs from the systems they manage:

1. **Enable logging** — OS (Windows Event Logs, Linux syslog), applications, network devices, databases, cloud platforms
2. **Configure log forwarding** — to the centralized logging/SIEM platform; set up agents/collectors
3. **Maintain log infrastructure** — collectors, pipelines, storage, performance, and availability
4. **Ensure system-level audit logging** — server access, admin activity, configuration changes
5. **Maintain retention** per policy

*Typical teams:* Cloud Operations, Infrastructure, DevOps, Platform Engineering, Network Team, Application Team

### Security Team
Defines what must be logged and analyzes the results:

1. **Define logging requirements** — which systems, which events, retention periods, compliance needs
2. **SIEM management** — architecture, parsing/normalization, correlation rules
3. **Threat detection** — alerting rules, detection engineering, behavioral analytics
4. **Security monitoring** — SOC monitoring, incident detection, threat hunting
5. **Compliance & audit** — log integrity, audit trail validation, regulatory requirements

*Typical teams:* SOC, Security Engineering, Detection Engineering, Threat Intelligence, Incident Response

### Recommended RACI

| Activity | Owner |
|---|---|
| Define logging standards | Security |
| Enable logging on systems | IT / Cloud / DevOps |
| Configure log forwarding | IT / DevOps |
| Maintain logging infrastructure | IT / Platform |
| SIEM architecture | Security |
| Log parsing and normalization | Security |
| Alert rules and detection | Security |
| Monitoring and investigation | SOC |
| Compliance validation | Security |

### Cloud Example (Azure / AWS)

- **Cloud Operations** enables logs such as Azure Activity Logs, Azure AD/Entra ID logs, firewall logs, NSG flow logs, VM logs, application logs, and load balancer logs.
- **Security** owns the SIEM (e.g., Sentinel, Splunk), detection rules, UEBA, threat hunting, and alert triage.

### Simple Rule of Thumb
Operations generates and ships logs. Security uses logs for detection and response.

### Common Mistake
Making Security responsible for the entire pipeline leads to missing logs, broken integrations, logging gaps, and poor monitoring. Best practice is shared ownership with a clear RACI.

**Short answer:** Log collection infrastructure → IT/Cloud Operations. Log requirements and monitoring → Security.

## 5. Log Classification and Retention Baseline

Logs generally fall into three tiers:

1. **Mandatory (security-critical)** — required for detection, forensics, and compliance
2. **Recommended/optional** — useful for deep investigation and threat hunting
3. **Not required for security** — mainly performance or operational logs

A common enterprise storage tiering model:

- **Hot** (SIEM-searchable): 30–90 days
- **Warm** (log archive): 180 days
- **Cold** (compliance storage): 1–2 years

### Windows OS Logs (Servers/Workstations)

**Mandatory**

| Log Type | Key Events | Retention |
|---|---|---|
| Security Event Logs | Logon/logoff (4624, 4625), privilege escalation (4672), account changes | 1 year |
| Account Management Logs | User creation/deletion, password changes | 1 year |
| Process Creation Logs (Sysmon/Event 4688) | Malware detection, suspicious process execution | 180 days |
| PowerShell Logs | Script execution, encoded commands | 180 days |
| Windows Defender Logs | Malware detection | 180 days |
| RDP Logs | Remote access activity | 1 year |

**Recommended:** Sysmon logs (advanced threat detection), Task Scheduler logs (persistence detection), service installation logs (malware persistence) — 180 days.

**Not required:** Print service logs, hardware device logs (operational only).

### Linux OS Logs

**Mandatory**

| Log | Location | Retention |
|---|---|---|
| Authentication logs | `/var/log/auth.log` or `/var/log/secure` | 1 year |
| SSH logs | SSH login attempts | 1 year |
| Sudo logs | Privilege escalation | 1 year |
| User account changes | `/etc/passwd` modifications | 1 year |
| Kernel logs | Suspicious kernel activity | 180 days |

**Recommended:** Auditd logs (detailed command auditing), process execution logs (threat detection) — 180 days.

**Not required:** Cron job success logs, package install logs (operational only).

### Azure VM Logs

**Mandatory:** Azure Activity Logs, VM guest OS logs, Entra ID sign-in logs, Defender for Cloud alerts — 1 year.

**Recommended:** VM diagnostic logs, boot diagnostics — 180 days.

**Not required:** VM performance metrics.

### Azure PaaS Logs

**Mandatory:** Azure Activity Logs, Azure AD audit logs, Azure AD sign-in logs — 1 year.

**Recommended:** Resource diagnostic logs, Defender for Cloud logs.

### Firewall Logs

**Mandatory:** Allow/deny traffic logs, admin activity logs, VPN access logs — 1 year.

**Recommended:** Threat intelligence matches (IOC detection), IDS/IPS logs (exploit detection).

**Not required:** Performance metrics.

### WAF Logs

**Mandatory:** Blocked requests (1 year), allowed-but-suspicious traffic (180 days), rule-triggered logs (1 year).

**Recommended:** Bot detection logs, geo-based blocking logs.

### Application Gateway Logs

**Mandatory:** Access logs (180 days), firewall logs if WAF enabled (1 year).

**Recommended:** Performance logs.

### Load Balancer Logs

**Mandatory:** Access logs — 180 days.

**Recommended:** Health probe logs.

**Not required:** Performance metrics.

### Key Vault Logs

**Mandatory (all 1 year):** Secret access logs, key operation logs, admin activity/access-policy changes.

### Database Logs (SQL/NoSQL)

**Mandatory (all 1 year):** Authentication logs, query audit logs (data exfiltration detection), admin activity (privilege escalation).

**Recommended:** Slow query logs, configuration change logs.

### Storage Account Logs

**Mandatory (all 1 year):** Blob access logs, file access logs, account key usage logs.

**Recommended:** Network access logs.

### Identity Logs (highest priority in cloud environments)

**Mandatory:** Entra ID sign-in logs (1–2 years), Entra ID audit logs (1–2 years), conditional access logs (1 year), privileged identity management logs (1 year).
